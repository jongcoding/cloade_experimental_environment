| 모드 | 진입점 | 설명 |

### 에이전트 조율 패턴

오케스트레이터는 상황에 따라 6가지 패턴으로 에이전트를 스폰한다:

| 패턴 | 상황 | 에이전트 흐름 |
|------|------|-------------|
| A. Stage 추가 | 새 Stage 구현 | TF Engineer → Executor(단독→연결→회귀) |
| B. 체인 설계 | 시나리오 초안 | Chain Architect → (NOVEL 시) 패턴 A |
| C. 탐색+추가 | 기법 불확실 | Recon Scout ∥ TF Engineer → 패턴 A |
| D. 회귀 실패 | 긴급 대응 | Failure Analyst → TF Engineer ∥ Executor |
| E. 중간 검증 | Stage 2-3 완성 | Chain Architect ∥ Executor (병렬) |
| F. 최종 완성 | 전체 완료 | Executor(3회) → Chain Architect ∥ Executor |

`→` 순차, `∥` 병렬

**상세 패턴 설명**: `.claude/skills/orchestrate/SKILL.md`

## 핵심 구조: 하네스 루프

이 프로젝트의 핵심은 **선형 파이프라인이 아니라 반복 루프**다.
"설계 → 구현 → 마지막에 검증"이 아니라, **한 조각을 만들 때마다 검증하고, 전체를 이을 때마다 다시 검증하는 구조**.

```
                    ┌──────────────────────────────┐
                    │                              │
                    ▼                              │
        ┌───────────────────┐                      │
        │  Stage N Terraform │ ◄── 수정             │
        │  작성 (한 조각)    │                      │
        └────────┬──────────┘                      │
                 │ terraform apply                  │
                 ▼                                  │
        ┌───────────────────┐     실패              │
        │  Stage N 단독 검증 │ ──────────┐          │
        │  "이 Stage만 공격  │           │          │
        │   가능한가?"       │           │          │
        └────────┬──────────┘           │          │
                 │ 성공                  │          │
                 ▼                      ▼          │
        ┌───────────────────┐   ┌──────────────┐   │
        │  Stage N-1 → N    │   │ 원인 분석     │   │
        │  연결 검증         │   │ Terraform 수정│   │
        │  "이전 결과물로    │   │ 또는 체인 수정│   │
        │   이 Stage 진입    │   └──────┬───────┘   │
        │   가능한가?"       │          │           │
        └────────┬──────────┘          └───────────┘
                 │ 성공                        ▲
                 ▼                             │
        ┌───────────────────┐     실패         │
        │  전체 체인 회귀 검증│ ────────────────┘
        │  Stage 0 → ... → N│
        │  처음부터 끝까지   │
        └────────┬──────────┘
                 │ 성공
                 ▼
        ┌───────────────────┐
        │  Stage N+1로 확장  │
        │  (다음 조각 추가)  │
        └───────────────────┘

    ※ 마지막 Stage까지 완성 후에도 전체 루프를 3회 반복
    ※ 매 반복마다 terraform destroy → apply → 전체 공격 체인 실행
```

## 루프 상세

### Loop 1: 단일 Stage 루프 (가장 빈번)

```
한 Stage의 Terraform을 작성한다
  ↓
terraform apply
  ↓
이 Stage의 공격을 실행한다 (AWS CLI / boto3)
  ↓
┌─ 성공 → 결과 기록 → 다음 검증으로
└─ 실패 → 에러 분석 → Terraform 수정 또는 공격 명령 수정 → 다시 apply
```

**이 루프의 빈도**: Stage당 3-10회
**이 루프에서 발견되는 것**: 개별 기법의 실제 동작, 필요 조건, 에러 패턴

### Loop 2: Stage 간 연결 루프 (핵심)

```
Stage N-1의 출력물(자격증명, 정보 등)을 가져온다
  ↓
그 출력물만으로 Stage N에 진입할 수 있는가?
  ↓
┌─ 성공 → 체인 연결 확인 → 다음 Stage 추가
└─ 실패 → 연결 고리 분석:
     ├─ 출력물이 부족한가? → Stage N-1의 Terraform/공격 수정
     ├─ Stage N의 진입 조건이 틀렸는가? → Stage N의 Terraform 수정
     └─ 중간에 빠진 Stage가 필요한가? → Stage N-0.5 신규 추가
```

**이 루프의 빈도**: 연결 지점당 2-5회
**이 루프에서 발견되는 것**: 체인의 실현 가능성, 빠진 단계, 예상치 못한 장벽

### Loop 3: 전체 체인 회귀 루프 (가장 중요)

```
terraform destroy (전체 정리)
  ↓
terraform apply (처음부터 배포)
  ↓
Stage 0부터 순서대로 공격 실행
  ↓
┌─ 전체 성공 → 회차 기록 → 3회 연속 성공할 때까지 반복
└─ 중간에 실패 → 실패 Stage 식별 → Loop 1 또는 Loop 2로 돌아감
```

**이 루프의 빈도**: 최소 3회 (완성 후)
**이 루프에서 발견되는 것**: 타이밍 이슈, 리소스 생성 순서 의존성, 재현성 문제

### Loop 4: 미끼 경로 검증 루프

```
풀이자 관점에서 미끼 경로를 시도한다
  ↓
미끼가 자연스럽게 발견되는가?
  ↓
미끼를 따라가면 적절한 시점에 막히는가? (너무 빨리/늦지 않게)
  ↓
막힌 후 올바른 경로로 돌아올 단서가 있는가?
  ↓
┌─ 자연스러움 → OK
└─ 부자연스러움 → 미끼 Terraform 수정 → 다시 시도
```

### Loop 5: 신규성 검증 루프 (외부 소스 포함)

```
[Phase 1] 내부 비교 — CloudGoat DB + 기존 풀이 시나리오
  ↓ 4항목 매트릭스 (진입점, 피봇, 권한상승, 최종도달)
  ↓ 유사도 ≤1/4 → OK
  ↓ 유사도 ≥2/4 → 체인 재설계 (STOP)
  ▼
[Phase 2] CTF 대회 비교 — Cloud Village(DEFCON), WIZ, HTB, flAWS
  ↓ Writeup 검색 + 웹 조사
  ↓ 동일 진입점+체인 발견 → 차별화 또는 재설계
  ▼
[Phase 3] 교육 플랫폼 비교 — AWSGoat, CloudFoxable, IAM Vulnerable 등
  ↓ GitHub/공식 문서 확인
  ↓ 동일 체인 발견 → 차별화 또는 재설계
  ▼
[Phase 4] 컨퍼런스/연구 비교 — fwd:cloudsec, DEFCON, BlackHat, 블로그
  ↓ 발표 목록 + 기법 블로그 검색
  ↓ 동일 공격 체인 발표 발견 → 차별화 필요
  ▼
[Phase 5] 개별 기법 vs 체인 분리 평가
  ↓ 각 기법: NOVEL / KNOWN / WELL-KNOWN / COMMON
  ↓ 체인 전체: 조합의 전례 유무
  ↓ 위험 요소 식별 + 대응 방안
  ▼
최종 판정: NOVEL / PARTIAL / REJECT
  → experiment_log/novelty_assessment.md에 보고서 작성
```

**실행 시점**: 초안 설계 시 (Phase 1 + 간략 검색), Stage 2-3 완성 시 (Phase 1-4), 최종 완성 시 (Phase 1-5 + 보고서)
**이 루프에서 발견되는 것**: 기존 CTF/교육 플랫폼과의 중복, 알려진 기법의 정확한 공개 수준, 체인 차별화 포인트

## 실제 세션 흐름

### 1회차 세션: Stage 0 (진입점)

```
[프롬프트: 01_system_prompt.md + 07_harness_loop.md]

1. Stage 0의 Terraform 작성 (취약 웹앱)
2. terraform apply
3. 웹 취약점 공격 시도
4. 성공할 때까지 Loop 1 반복
5. Stage 0 완성 → 결과 기록
```

### 2회차 세션: Stage 1 (정찰) + 연결 검증

```
[프롬프트: 07_harness_loop.md, 이전 세션 결과 포함]

1. Stage 1의 Terraform 추가 (기존 Stage 0에 덧붙임)
2. terraform apply
3. Stage 1 단독 검증 (Loop 1)
4. Stage 0 → 1 연결 검증 (Loop 2)
   - Stage 0에서 얻은 자격증명으로 Stage 1 진입 가능한가?
5. 문제 있으면 수정 후 재검증
6. Stage 0→1 완성 → 결과 기록
```

### 3회차 세션: Stage 2 (래터럴) + 회귀 검증

```
1. Stage 2의 Terraform 추가
2. terraform apply
3. Stage 2 단독 검증 (Loop 1)
4. Stage 1 → 2 연결 검증 (Loop 2)
5. [중요] Stage 0 -> 1 -> 2 전체 회귀 검증 (Loop 3)
   - 처음부터 다시! Stage 2 추가로 Stage 0이나 1이 깨지지 않았는가?
6. 문제 있으면 해당 Stage로 돌아가서 수정
```

### N회차 세션: 반복...

```
매 Stage 추가 시:
1. 새 Stage 단독 검증 (Loop 1)
2. 이전 Stage → 새 Stage 연결 검증 (Loop 2) 
3. 전체 체인 회귀 검증 (Loop 3)     ← 이게 매번 들어가야 함

Stage 2-3 완성 시 (중간 체크포인트):
4. 신규성 검증 Phase 1-4 실행 (Loop 5)  ← 외부 CTF/플랫폼/컨퍼런스 비교
5. PARTIAL/REJECT 판정 시 체인 수정 후 Loop 1-3 재실행

전체 완성 후:
6. terraform destroy → apply → 전체 공격 × 3회 (Loop 3)
7. 미끼 경로 검증 (Loop 4)
8. 최종 신규성 검증 Phase 1-5 + 보고서 (Loop 5)  ← novelty_assessment.md 작성
9. 난이도 조정
```

하네스 루프에서 가장 중요한 것은 **어디까지 검증됐는지** 추적하는 것이다.

### harness_state.md (매 세션 끝에 업데이트)

```markdown
# Harness State -- [시나리오 코드네임]
마지막 업데이트: [날짜]

## 완성된 Stage
| Stage | Terraform | 단독 검증 | 연결 검증 | 회귀 검증 | 비고 |
|-------|-----------|-----------|-----------|-----------|------|
| 0     | DONE      | PASS (3/3)| -         | PASS (3/3)|      |
| 1     | DONE      | PASS (3/3)| PASS 0->1 | PASS (2/3)| 3회차에서 SG 타이밍 이슈 |
| 2     | DONE      | PASS (2/3)| FAIL 1->2 | -         | 연결 실패: 크레덴셜 형식 불일치 |
| 3     | WIP       | -         | -         | -         | 작성 중 |
| 4     | TODO      | -         | -         | -         | 미착수 |
| 5     | TODO      | -         | -         | -         | 미착수 |

## 미끼 경로
| 미끼 | 구현 | 자연스러움 검증 | 데드엔드 확인 |
|------|------|----------------|--------------|
| Decoy 1 | DONE | FAIL 너무 뻔함 | - |
| Decoy 2 | WIP  | -              | - |

## 알려진 이슈
- [ ] Stage 1→2 연결에서 STS 토큰 만료 문제 (15분)
```

## 팁

1. **한 세션에 Stage 1-2개씩**: 너무 많으면 회귀 검증이 복잡해진다
2. **terraform destroy를 두려워하지 마라**: 하네스의 가치는 "언제든 처음부터 재현 가능"에 있다
3. **실패 로그를 정확히 남겨라**: "안 됐다"가 아니라 에러 메시지 전문을 기록
4. **회귀 검증을 건너뛰지 마라**: Stage 3에서 발견한 문제가 Stage 1의 Terraform 수정을 요구할 수 있다
5. **harness_state.md를 항상 최신으로**: 이게 Claude에게 주는 컨텍스트의 질을 결정한다
6. **멀티에이전트에서 병렬 활용**: 독립적 작업(정찰 + 인프라 확인)은 병렬로 스폰해서 시간 절약
7. **에이전트 결과 핵심만 전달**: 에이전트 간 데이터 전달 시 필요한 부분만 추출해서 컨텍스트 절약

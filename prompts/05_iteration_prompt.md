# 보조 프롬프트 -- 실패/발견 기반 심화 탐색

하네스 루프 중 단독/연결/회귀 검증 실패의 원인을 못 찾을 때, 또는 예상치 못한 결과를 깊이 파고들 때 사용한다.
분석이 끝나면 반드시 하네스 루프(07_harness_loop.md)로 복귀한다.

---

## 실패 심화 분석 프롬프트

```
## 실패 심화 분석

### 실패한 실험
- 실험 ID: [EXP-XXX]
- 시도한 것: [무엇을 했는지]
- 기대한 결과: [무엇을 예상했는지]
- 실제 결과: [무엇이 나왔는지]
- 에러 메시지: 
```
[에러 메시지 전문 붙여넣기]
```

### 분석해줘

1. **에러 원인 분석**: 이 에러가 정확히 왜 발생하는가?
   - IAM 권한 부족인가? → 어떤 권한이 필요한가?
   - 리소스 조건인가? → 어떤 조건이 충족되어야 하는가?
   - 네트워크 제한인가? → 어떤 경로가 차단되었는가?
   - 서비스 자체의 제한인가? → 이 제한은 우회 가능한가?

2. **우회 가능성 탐색**: 이 제한을 우회할 수 있는 방법이 있는가?
   - 다른 API 엔드포인트로 같은 결과를 얻을 수 있는가?
   - 다른 서비스를 경유해서 접근할 수 있는가?
   - 시간/순서를 바꾸면 되는가?
   - 조건을 변경할 수 있는가? (태그 변경, 설정 수정 등)

3. **시나리오 활용**: 이 실패를 시나리오의 난이도 요소로 활용할 수 있는가?
   - "이 공격은 직관적으로 보이지만 실패한다 → 풀이자가 왜 실패하는지 이해해야 다음 단계로 넘어갈 수 있다"
   - 이 실패 자체가 미끼/데드엔드가 될 수 있는가?

4. **추가 실험 제안**: 이 분석을 기반으로 어떤 추가 실험을 해야 하는가?
   - 우회 경로 실험
   - 조건 변경 실험
   - 관련 서비스 조합 실험

직접 추가 실험을 설계하고 실행해줘.
```

## 예상치 못한 성공 심화 프롬프트

```
## 예상치 못한 발견 심화

### 발견 사항
- 실험 ID: [EXP-XXX]
- 시도한 것: [무엇을 했는지]
- 예상한 결과: [무엇을 기대했는지]
- **실제 결과**: [예상과 다르게 성공한 것 또는 추가로 알게 된 것]

### 심화 분석

1. **왜 이게 가능한가?**
   - AWS 문서에 이 동작이 명시되어 있는가?
   - 이것은 의도된 동작인가, 버그인가?
   - 어떤 조건에서만 가능한가? (리전, 서비스 버전, 계정 설정 등)

2. **이것을 공격에 활용할 수 있는가?**
   - 이 동작으로 어떤 자격증명/정보를 얻을 수 있는가?
   - 이 동작의 영향 범위는? (단일 리소스? 계정 전체?)
   - 이 동작을 탐지할 수 있는가? (CloudTrail 로그에 남는가?)

3. **재현 가능한가?**
   - 이 결과를 100% 재현할 수 있는 최소 조건을 정리해줘
   - Terraform으로 이 환경을 자동 배포할 수 있는가?
   - 타이밍 이슈가 있는가? (Eventually Consistent 등)

4. **시나리오에 통합**
   - 이 발견을 어떤 Stage에 배치하면 좋겠는가?
   - 이 발견을 모르면 풀 수 없는 구조로 만들 수 있는가?
   - 이 발견에 대한 힌트는 어떻게 제공할 것인가?

심화 실험을 설계하고 실행해줘.
```

## 서비스 간 상호작용 탐색 프롬프트

```
## 서비스 간 상호작용 탐색

### 탐색 대상
서비스 A: [예: Lambda]
서비스 B: [예: Step Functions]
연결 방식: [예: Step Functions가 Lambda를 호출]

### 탐색 질문

이 두 서비스가 상호작용할 때, 다음을 실험으로 확인해줘:

1. **서비스 A의 자격증명으로 서비스 B를 제어할 수 있는가?**
   - A의 역할로 B의 설정을 변경할 수 있는가?
   - A에서 B로 자격증명이 전달되는가?
   - A와 B 사이에 Trust 관계가 있는가?

2. **서비스 B가 서비스 A를 호출할 때 어떤 자격증명을 사용하는가?**
   - B의 실행 역할인가?
   - A에 전달된 역할인가?
   - 서비스 연결 역할(Service-Linked Role)인가?

3. **이 상호작용을 가로챌 수 있는가?**
   - 중간에 끼어들어서 요청/응답을 변조할 수 있는가?
   - 대상을 변경할 수 있는가? (다른 Lambda를 호출하도록)
   - 이벤트를 위조할 수 있는가?

4. **이 상호작용에서 민감 정보가 노출되는가?**
   - 로그에 자격증명이 기록되는가?
   - 환경변수에 정보가 전달되는가?
   - 임시 파일/캐시에 정보가 남는가?

각 질문에 대해 가설을 세우고, 최소 Terraform 환경을 만들고, 실험해줘.
```

## 크로스커팅 탐색 프롬프트 (여러 기법 조합)

```
## 크로스커팅 실험: 기법 조합

### 조합할 기법들
- 기법 A: [실험으로 검증된 기법 — 예: "ECS Task Def Backdoor"]
- 기법 B: [실험으로 검증된 기법 — 예: "Tag-based ABAC Bypass"]  
- 기법 C: [실험으로 검증된 기법 — 예: "SSM Session via Cross-Role"]

### 질문
이 3개 기법을 하나의 연속 체인으로 연결할 수 있는가?

구체적으로:
1. 기법 A의 결과물(output)이 기법 B의 입력(input)이 될 수 있는가?
2. 기법 B의 결과물이 기법 C의 입력이 될 수 있는가?
3. 이 체인에서 빠진 연결 고리는 무엇인가?
4. 빠진 고리를 채울 수 있는 기법/서비스가 있는가?

### 실험
1. 기법 A→B 연결을 실험해줘
2. 기법 B→C 연결을 실험해줘
3. 전체 A→B→C 체인을 한 번에 실행해줘
4. 체인이 깨지는 지점이 있으면 분석해줘

### 결과물
- 연결 가능/불가능 판정
- 필요한 추가 조건
- 최종 체인 다이어그램
```

### 작업 규칙 (매 세션 반드시 준수)

**규칙 1: 한 조각씩 만들고 즉시 검증**
Terraform을 Stage 전체가 아니라 리소스 1-2개 단위로 추가하고, 
추가할 때마다 terraform apply + 해당 부분 공격 테스트.
한 번에 큰 덩어리를 만들고 나중에 테스트하지 마라.

**규칙 2: 3단계 검증을 항상 수행**
새 Stage를 추가하면 반드시 순서대로:
  ① 단독 검증: 이 Stage만 공격 가능한가?
  ② 연결 검증: 이전 Stage 결과물로 이 Stage 진입 가능한가?
  ③ 회귀 검증: Stage 0부터 여기까지 전체가 여전히 작동하는가?

셋 중 하나라도 실패하면 다음 Stage로 넘어가지 마라.

**규칙 3: 매 검증 결과를 구조화해서 기록**
```
### 검증 결과: Stage [N] — [날짜/시간]

#### ① 단독 검증
- 명령어: [실행한 명령어]
- 결과: PASS / FAIL
- 출력: [전체 출력]
- 소요시간: [분]

#### ② 연결 검증 (Stage N-1 -> N)
- 입력: [이전 Stage에서 가져온 것]
- 명령어: [실행한 명령어]  
- 결과: PASS / FAIL
- 출력: [전체 출력]

#### ③ 회귀 검증 (Stage 0 -> N)
- terraform destroy: PASS / FAIL
- terraform apply: PASS / FAIL
- Stage 0: PASS / FAIL
- Stage 1: PASS / FAIL
- ...
- Stage N: PASS / FAIL
- 총 소요시간: [분]
```

**규칙 4: 실패 시 즉시 분석하고 수정**
실패하면:
1. 에러 메시지 전문 기록
2. 원인 분류:
   - Terraform 문제 (리소스 설정 오류)
   - IAM 권한 문제 (정책이 너무 좁거나 넓음)
   - 네트워크 문제 (SG, 라우팅, 서브넷)
   - 타이밍 문제 (리소스 생성/전파 지연)
   - 공격 명령어 문제 (잘못된 API 사용)
3. 수정 적용
4. 같은 검증 다시 실행
5. 이 수정-재검증 루프를 성공할 때까지 반복

**규칙 5: 검증 자동화 스크립트를 점진적으로 구축**
수동으로 검증이 성공하면, 그 명령어를 즉시 스크립트로 만들어라.
```bash
# validation/test_stage_N.sh
#!/bin/bash
set -e
echo "=== Stage N: [이름] ==="
# [수동으로 성공한 명령어를 그대로 스크립트화]
echo "PASS: Stage N"
```
다음 회귀 검증 때는 이 스크립트를 사용한다.
이렇게 하면 Stage가 쌓일수록 회귀 검증이 자동화된다.

### 세션 시작 시 할 것

1. 현재 Terraform 상태 확인:
   ```bash
   cd terraform/
   terraform plan  # 드리프트 없는지 확인
   ```

2. 이전 세션의 인프라가 살아있으면:
   - 바로 이어서 작업 (terraform apply는 이미 됨)
   
3. 이전 세션에서 destroy했으면:
   ```bash
   terraform apply -auto-approve
   # 기존 검증 스크립트로 baseline 확인
   bash validation/test_stage_0.sh
   bash validation/test_stage_1.sh
   # ...현재까지 완성된 Stage 전부
   ```

### 세션 종료 시 할 것

1. harness_state.md 업데이트
2. 이번 세션의 발견/변경 기록
3. 다음 세션에서 할 것 명시
4. 비용 확인: `aws ce get-cost-and-usage --time-period Start=YYYY-MM-DD,End=YYYY-MM-DD`
5. 선택: terraform destroy (비용 절감) 또는 유지 (다음 세션에서 바로 이어서)

---

자 시작하자. 현재 하네스 상태를 보고 이번 세션의 작업을 진행해줘.
검증은 반드시 ① 단독 → ② 연결 → ③ 회귀 순서로.
```

## 변형: 첫 세션용 (Stage 0부터 시작)

```
## 하네스 루프 세션 — 첫 시작

시나리오: [코드네임]
AWS 계정: [계정 ID]
리전: us-east-1

### 공격 체인 설계 (초안)

[이미 설계가 있으면 붙여넣기, 없으면:]

다음 요소를 포함하는 5단계 이상의 공격 체인을 설계해줘:
- Stage 0: 웹 취약점 진입점
- Stage 1-4: 정찰 → 래터럴 → 크레덴셜 → 권한상승
- Stage 5: Flag
- 미끼: 2개 이상

설계는 간략히. 단, 설계 후 반드시 **신규성 검증**을 해라:
REFERENCE.md의 기존 시나리오 DB와 비교해서 체인이 기존과 다른지 확인.
NOVEL 판정이 나와야 Stage 0 구현으로 넘어간다.

### Stage 0부터 시작

1. Stage 0의 최소 Terraform을 작성해줘
   - 취약한 웹앱 (Docker + ECS 또는 EC2)
   - 필요한 네트워크 (VPC, SG)
   - 웹 취약점 1개 (SSRF 또는 Command Injection)

2. terraform apply

3. 웹 취약점 공격 검증
   - 취약점이 동작하는가?
   - 취약점으로 어떤 정보/접근을 얻을 수 있는가?

4. 검증 결과 기록 + 검증 스크립트 생성

5. Stage 0 완성 → Stage 1 추가

규칙: 한 조각씩 만들고 즉시 검증. 검증 안 된 코드 위에 다음 Stage 올리지 마라.
```

## 변형: 회귀 실패 대응용

```
## 회귀 실패 긴급 대응

### 실패 상황
- 회귀 검증 범위: Stage 0 → Stage [N]
- 실패 지점: Stage [X]
- 에러: 
```
[에러 메시지 전문]
```

### 이전에는 작동했는가?
- Stage [X] 단독 검증: 마지막으로 성공한 시점 [날짜]
- 그 이후 변경된 것: [Stage Y의 Terraform 수정, 새 Stage Z 추가 등]

### 분석해줘

1. **변경 영향 분석**: 마지막 성공 이후 변경된 것 중, Stage [X]에 영향을 줄 수 있는 것은?
   - IAM 정책 변경이 다른 Stage에 영향?
   - SG 규칙 변경이 네트워크에 영향?
   - 리소스 이름/태그 변경이 참조에 영향?

2. **격리 테스트**: 문제를 격리하기 위해:
   - Stage [X]만 단독으로 다시 검증
   - Stage [X-1] → [X] 연결만 다시 검증
   - 변경 전 Terraform으로 롤백해서 다시 검증

3. **수정 적용**: 원인을 찾으면 수정하고, 전체 회귀를 다시 돌려줘.

수정할 때 주의: Stage [X]만 고치면 Stage [X+1] 이후가 깨질 수 있으니,
수정 후 반드시 Stage 0 → [N] 전체를 다시 검증해줘.
```

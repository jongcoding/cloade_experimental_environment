# 보조 프롬프트 -- 미지 영역 탐색 (Novel Discovery)

체인에 넣을 기법이 부족하거나, 기존 CloudGoat에 없는 새 공격 벡터가 필요할 때 사용한다.
유망한 벡터를 발견하면 반드시 하네스 루프(07_harness_loop.md)로 복귀해서 Stage로 구현한다.

---

## 전략 1: 새로운 AWS 서비스의 공격 표면 탐색

```
## 신규 서비스 공격 표면 탐색

### 대상 서비스
서비스명: [예: Amazon Bedrock AgentCore, Amazon Q, AWS App Runner 등]
출시 시기: [최근 1년 이내 서비스 우선]
AWS 문서 URL: [해당 서비스의 IAM 액션 문서]

### 탐색 방법

이 서비스에 대해 아래 순서로 공격 표면을 탐색해줘:

**1단계: IAM 액션 열거**
```bash
# 이 서비스의 모든 IAM 액션 목록을 가져와줘
# AWS Policy Generator 또는 문서에서 확인
```

다음을 분류해줘:
- 읽기 전용 액션 (Get*, List*, Describe*)
- 변경 액션 (Create*, Update*, Delete*, Put*)
- 실행 액션 (Invoke*, Start*, Run*)
- 위험 액션 (PassRole, AssumeRole 관련)

**2단계: Trust Policy 분석**
- 이 서비스의 Service Principal은 무엇인가?
- 다른 서비스와 Service Principal을 공유하는가?
- → agentcore_identity_confusion처럼 Identity Confusion 가능성

**3단계: 자격증명 전달 경로**
- 이 서비스는 실행 역할(Execution Role)을 사용하는가?
- 실행 역할의 자격증명은 어디에서 접근 가능한가?
  - IMDS? Container Credentials API? 환경변수?
- 이 자격증명을 탈취할 수 있는 경로가 있는가?

**4단계: 실험**
가장 유망한 공격 벡터 1-2개를 선택해서 실험해줘.
최소 Terraform → 배포 → 공격 시도 → 결과 기록.

### 결과물
- 이 서비스의 공격 표면 요약
- 실험 결과 (성공/실패)
- CloudGoat 시나리오로 만들 수 있는 가능성 평가
```

## 전략 2: 알려진 기법의 경계 조건 탐색

```
## 경계 조건 실험

### 알려진 기법
기법: [예: "iam:PassRole로 다른 역할에 자격증명 전달"]
출처: [CloudGoat 시나리오명 또는 문서]

### 이 기법의 "당연하다고 가정하는 것들"

이 기법이 작동하려면 다음 조건이 필요하다고 알려져 있다:
1. [조건 1 — 예: "PassRole 권한이 있어야 한다"]
2. [조건 2 — 예: "대상 역할의 Trust Policy가 해당 서비스를 허용해야 한다"]
3. [조건 3 — 예: "..."]

### 실험할 경계 조건

위 조건들의 경계를 실험으로 탐색해줘:

1. **조건 완화 실험**: 조건 N이 없어도 기법이 작동하는가?
   - 예: PassRole 없이 다른 방법으로 역할을 전달할 수 있는가?
   - 예: Trust Policy에 Condition이 있으면 정말 차단되는가?

2. **조건 우회 실험**: 조건 N을 직접 충족시키지 않고 우회할 수 있는가?
   - 예: 직접 PassRole은 못하지만, 다른 서비스가 대신 전달해주는가?
   - 예: SCP가 있으면 진짜 차단되는가? SCP의 빈틈은 없는가?

3. **조건 조합 실험**: 여러 "약한" 조건의 조합으로 기법이 작동하는가?
   - 예: PassRole은 없지만 CreateRole + AttachRolePolicy가 있으면?
   - 예: 읽기 전용 권한 3-4개의 조합으로 쓰기 효과를 낼 수 있는가?

4. **시간/순서 실험**: 실행 순서에 따라 결과가 달라지는가?
   - 예: 역할을 먼저 만들고 나중에 정책을 붙이면?
   - 예: Eventually Consistent한 IAM 전파 시간을 활용할 수 있는가?

### 실험 형식
각 실험을 다음 형식으로 기록:
- 가설: "조건 X가 없으면 기법 Y는 실패할 것이다"
- 실험: [Terraform + 명령어]
- 결과: [성공/실패 + 출력]
- 발견: [문서와 다른 점, 새로운 사실]
```

## 전략 3: 에러 메시지 기반 정보 추출

```
## 에러 메시지 기반 탐색

### 배경
AWS API의 에러 메시지는 종종 의도치 않은 정보를 노출한다.
권한이 없는 API를 호출했을 때의 에러 메시지가 "리소스가 존재하는지"를 알려주는 경우가 대표적이다.

### 실험

주어진 IAM 자격증명으로 다음 패턴의 에러 메시지를 수집해줘:

1. **존재하는 리소스 vs 존재하지 않는 리소스에 대한 에러 차이**
```bash
# 존재하는 역할
aws iam get-role --role-name existing_role 2>&1
# 존재하지 않는 역할
aws iam get-role --role-name nonexistent_role 2>&1
# 에러 메시지가 다른가? → 리소스 존재 여부를 알 수 있는가?
```

2. **권한 부족 에러에서 노출되는 정보**
```bash
# 다양한 서비스의 권한 부족 에러를 수집
aws s3 ls 2>&1
aws ec2 describe-instances 2>&1
aws lambda list-functions 2>&1
# 에러 메시지에 계정 ID, ARN, 역할 이름 등이 포함되어 있는가?
```

3. **조건부 정책의 에러에서 조건 정보 노출**
```bash
# Condition이 있는 정책에 대해
# 조건을 충족하지 못했을 때의 에러 메시지가 어떤 조건이 필요한지 알려주는가?
```

### 시나리오 활용
에러 메시지에서 정보를 추출하는 기법을 시나리오의 정찰 단계에 넣을 수 있다.
풀이자가 에러 메시지를 꼼꼼히 읽어야만 다음 단계의 단서를 얻도록 설계.
```

## 전략 4: CloudTrail 로그를 활용한 역방향 설계

```
## CloudTrail 역방향 설계

### 개념
정상적인 AWS 운영에서 발생하는 CloudTrail 로그를 분석해서,
"이 로그가 있다면 어떤 공격이 가능한가?"를 역방향으로 설계한다.

### 실험

1. Terraform으로 일반적인 인프라를 배포 (웹앱 + DB + CI/CD)
2. 정상 운영 시 발생하는 CloudTrail 이벤트를 수집
3. 각 이벤트에 대해:
   - "이 API를 공격자가 호출하면 무엇을 얻을 수 있는가?"
   - "이 API의 요청/응답에 민감 정보가 포함되어 있는가?"
   - "이 API를 호출한 주체의 권한으로 다른 무엇을 할 수 있는가?"

### 목표
- 방어자 관점에서 모니터링하기 어려운 공격 경로 발견
- "정상 트래픽처럼 보이는 공격" 시나리오 설계
- CloudTrail에 로그가 남지 않는 동작 발견 (Data Events 제외 등)
```

## 메타 프롬프트: 실험 결과 종합 & 다음 방향 결정

```
## 실험 결과 종합

### 지금까지의 실험 결과

[여기에 모든 실험 결과를 붙여넣기 — experiment_log/에서]

### 분석 요청

1. **가장 유망한 공격 벡터 Top 3**
   - 실험으로 검증됨 + 시나리오에 활용 가능 + 기존 CloudGoat에 없음

2. **추가 실험이 필요한 영역 Top 3**
   - 부분적으로 성공했지만 추가 조건 확인 필요
   - 또는 실패했지만 우회 가능성 있음

3. **가장 어려운 시나리오를 만들 수 있는 조합**
   - 검증된 기법들을 5단계 이상 체인으로 조합
   - 미끼 경로 포함
   - "이런 시나리오는 본 적 없다"고 말할 수 있는 수준

4. **다음 세션에서 해야 할 것**
   - 추가 실험? 체인 설계? 하네스 루프로 Stage 구현?
   - 구체적인 다음 프롬프트 제안

분석해줘.
```
